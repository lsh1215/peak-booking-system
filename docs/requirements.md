# 추출 요구사항

> 원문 문서에서 특정 조직명, 외부 절차, 일정, 연락처 등 공개 저장소에 드러나면 안 되는 맥락을 제거하고, 구현에 필요한 요구사항만 원문에 가깝게 추출한 문서입니다.

## 기술 요구사항

### 기술 스택

| 구분 | 요구사항 |
|---|---|
| 언어 | Java 8 이상, 혹은 Kotlin |
| 프레임워크 | Spring Boot 2.7 이상 |
| RDB | MySQL 또는 그 계열(MariaDB) |
| Cache | Redis |
| Infra | 애플리케이션 서버 2대 이상의 분산 환경을 가정 |
| Other | 필요에 따라 자유롭게 선택 가능. 단, 도입 근거를 `DECISIONS.md`에 필수 기재 |

## 요구 사항

### 주제

특정 시간(`00시`)에 오픈되는 **초특가 숙소 상품 `10개 한정`**에 대한 선착순 예약 시스템을 구축한다.

2대 이상의 애플리케이션 서버로 구성된 분산 환경을 가정하며, 다음 요건을 충족해야 한다.

- 모든 사용자에게 동등한 기회를 제공한다.
- 엄격한 재고 정합성을 유지한다.
- 결제 실패 및 시스템 장애 상황에서도 안정적으로 동작한다.

### 필수 구현 API

요청/응답 규격은 자유롭게 설계한다.

#### GET Checkout API

- 주문서 진입 API
- 상품 정보(명칭, 가격, 입/퇴실 시간 등) 및 사용자의 가용 Y포인트 등을 조회하는 API

#### POST Booking API

- 결제 및 예약 완료 API
- 주문서 정보를 입력받아 결제를 진행하고 최종 주문을 생성하는 API

## 핵심 요구 사항

아래 기능을 모두 구현한다.

### 1. 재고 정합성 및 공정성

- 대상 초특가 숙소 상품의 재고는 `10개`로 제한된다.
- `00시` 트래픽 집중 상황에서 미달되거나 초과판매가 발생하지 않도록 정합성을 완벽히 보장한다.
- 모든 사용자가 동등한 확률로 상품을 구매할 수 있는 구조를 고민한다.

### 2. 고가용성

- TPS 급증 대응을 대비하여 시스템 붕괴를 막기 위한 구조를 반영하고 `DECISIONS.md`에 기술한다.
- 평시 `50TPS`이며, 프로모션이 시작되는 `00시`에 약 `1~5분`간 순간적으로 `500~1000TPS`가 예상된다.

### 3. 멱등성 처리

- 주문서에서 아주 짧은 간격으로 연속되어 결제요청이 되어도 중복처리가 되지 않도록 멱등성 처리가 적용되어야 한다.

### 4. 결제 확장성

- 결제 수단: 신용카드, `Y페이`, `Y포인트`를 지원한다.
- 복합 결제: `신용카드 + Y포인트` 또는 `Y페이 + Y포인트` 가능.
- 단, 신용카드와 `Y페이`는 혼용 불가하다.
- 향후 새로운 결제 수단이 추가되어도 `Booking API`의 비즈니스 로직 수정을 최소화할 수 있는 구조를 적용하고 `DECISIONS.md`에 상세히 기술한다.

### 5. 장애 대응 및 예외 처리

- Redis 장애: Redis 장애 시의 Fallback 전략을 수립하고 근거를 제시한다.
- 결제 실패: 한도 초과 등 결제 실패 케이스에 대한 대응 로직을 설계한다.
- 해당 내용은 `DECISIONS.md`에 상세히 기술한다.

## 참고 사항

- 실제 PG사와의 연동은 생략하되, 인터페이스와 Mock 구현 등을 통해 실제 PG 결제 승인/조회/취소 또는 웹훅 흐름과 유사하게 구조적으로 이어지도록 작성한다.
- 회원 인증 및 로그인 보안 처리는 구현 범위에서 제외하므로 생략 가능하다.
- 인프라 증설(Scale-up/out)이 제한적인 상황을 가정한다.

## 산출물 요구사항

### 1. 동작 가능한 소스 코드

- 코드 수정 없이 실행이 가능해야 한다.
- 실행하기 위해 추가 인프라 구성이 필요한 경우는 `README.md`에 그 내용을 상세히 명시한다.

### 2. README.md

- 시스템 아키텍처 설명 및 실행 방법
- 시퀀스 다이어그램(Sequence Diagram) 또는 플로우차트 포함
- ERD 또는 DDL 스크립트(주문/결제 도메인 중심)

### 3. DECISIONS.md

- 주요 기술적 쟁점에 대한 선택 근거
- 라이브러리 도입 사유 및 문제 해결 전략 설명

## DECISIONS.md 작성 가이드

- 설계 과정에서 고민했던 기술적 판단, 트레이드오프, 선택의 근거를 자유로운 형식으로 기술한다.
- 어떤 주제를 쟁점으로 다룰지도 작성자의 판단에 맡긴다.
- 추가 인프라를 사용했다면 그 이유와 비용 대비 효과를 포함한다.

참고용 형식:

```md
## 쟁점 1. [제목]

### 상황

어떤 문제 상황이었는지

### 선택지

무엇을 선택했는지

### 왜 그렇게 판단했는지

어떤 대안이 있었고, 각각의 트레이드오프는 무엇이었는지
최종적으로 왜 이 방향을 선택했는지
```
