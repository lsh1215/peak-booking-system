# Requirements

> 제한 재고 예약/결제 백엔드 구현을 위한 공개 안전 요구사항 문서입니다. 원문에서 특정 조직, 평가 절차, 일정, 전달 방식, 특정 결제 브랜드명 등 외부 맥락을 식별할 수 있는 정보는 제거했습니다.

## 0. Project Baseline

이 프로젝트의 구현 기준은 다음과 같이 둔다.

| 영역 | 선택 |
|---|---|
| 언어 | Java 21 |
| 프레임워크 | Spring Boot 3.5.x |
| RDB | MySQL 8 |
| Cache | Redis |
| 실행 형태 | 2대 이상의 stateless application replica를 가정 |

원문 요구사항의 최소 기술 조건보다 높은 버전을 선택하되, 선택 근거와 트레이드오프는 `docs/decisions/DECISIONS.md`에 기록한다.

## 1. 문제 정의

특정 시간인 `00:00`에 오픈되는 한정 수량 숙소 상품에 대해 선착순 예약 시스템을 구축한다.

핵심 시나리오:

- 예약 가능한 상품 수량은 `10개`로 제한된다.
- 오픈 시각 직후 짧은 시간 동안 트래픽이 집중된다.
- 애플리케이션 서버는 `2대 이상`으로 구성된 분산 환경을 가정한다.

시스템은 다음 조건을 만족해야 한다.

- 모든 사용자에게 동등한 구매 기회를 제공한다.
- 재고 정합성을 엄격하게 유지한다.
- 결제 실패 및 시스템 장애 상황에서도 안정적으로 동작한다.

## 2. 기술 요구사항

## 3. 필수 구현 API

요청/응답 규격은 자유롭게 설계할 수 있다.

### 3.1 GET Checkout API

주문서 진입 시 필요한 정보를 조회하는 API다.

조회 대상 예시:

- 상품명
- 상품 가격
- 입실/퇴실 시간
- 사용자의 가용 포인트
- 예약/결제에 필요한 기타 주문서 정보

### 3.2 POST Booking API

주문서 정보를 입력받아 결제를 진행하고 최종 예약 또는 주문을 생성하는 API다.

반드시 고려해야 할 사항:

- 재고 정합성
- 결제 수단 검증
- 결제 실패 처리
- 멱등성 처리
- 중복 예약 방지

## 4. 핵심 요구사항

### 4.1 재고 정합성 및 공정성

`00:00` 트래픽 집중 상황에서도 재고가 미달되거나 초과 판매되지 않도록 정합성을 보장해야 한다.

필수 동작:

- 최종 확정 예약 수는 전체 재고 `10개`를 초과하면 안 된다.
- 결제 실패 후 재고가 영구히 잠기거나 누락되면 안 된다.
- 재고 상태 불일치로 인한 초과 판매와 미달 판매를 모두 방지해야 한다.
- 모든 사용자가 동등한 확률로 상품을 구매할 수 있는 구조를 고민해야 한다.

설계상 주의점:

- 단일 애플리케이션 인스턴스의 로컬 메모리나 로컬 락만으로는 분산 환경의 정합성을 보장할 수 없다.
- 중복 클릭이나 재시도가 특정 사용자의 성공 확률을 높이는 수단이 되면 안 된다.
- 한 사용자가 같은 한정 상품에 대해 여러 건의 최종 예약을 생성할 수 없어야 한다.

### 4.2 고가용성 및 순간 트래픽 대응

예상 트래픽:

- 평시: 약 `50 TPS`
- 오픈 직후 피크: 약 `500~1000 TPS`
- 피크 지속 시간: 약 `1~5분`

필수 동작:

- 순간적인 TPS 급증으로 시스템이 붕괴하지 않도록 해야 한다.
- 인프라 증설이 제한적인 상황을 가정하므로, 단순한 용량 증설에만 의존하면 안 된다.
- 과부하 상황에서도 예측 가능한 응답과 안정적인 실패 처리가 가능해야 한다.

설계에서 다룰 주제:

- Admission control
- 품절, 중복 요청, 과부하 요청에 대한 빠른 거절
- 애플리케이션 스레드, Redis 커넥션, DB 커넥션에 대한 backpressure
- 커넥션 풀 크기와 timeout 설정
- retry 정책과 retry storm 방지
- 동시 사용자 수뿐 아니라 목표 요청률 기반의 부하 테스트

### 4.3 멱등성 처리

주문서에서 매우 짧은 간격으로 결제 요청이 연속 발생해도 중복 처리가 되지 않아야 한다.

필수 동작:

- 동일한 멱등성 키와 동일한 요청 본문으로 재요청하면 동일한 논리 결과를 반환해야 한다.
- 동일한 멱등성 키로 서로 다른 요청 본문이 들어오면 충돌로 거절해야 한다.
- 중복 요청으로 인해 중복 예약 또는 중복 결제가 발생하면 안 된다.

설계에서 다룰 주제:

- `Idempotency-Key`의 범위
- 요청 본문 hash 비교
- 처리 결과 저장 및 재응답
- 장시간 `PROCESSING` 상태에 대한 timeout/recovery 정책
- 최종 방어선으로서 DB unique constraint

### 4.4 결제 확장성

다음 결제 수단을 지원해야 한다.

- 신용카드
- 간편결제
- 포인트

허용되는 결제 조합:

- 신용카드 + 포인트
- 간편결제 + 포인트
- 신용카드 단독
- 간편결제 단독
- 포인트 잔액이 충분한 경우 포인트 단독

허용되지 않는 결제 조합:

- 신용카드 + 간편결제
- 신용카드 + 간편결제 + 포인트
- 음수 금액
- 주문 금액과 결제 수단별 금액 합계 불일치
- 동일 결제 수단 중복 입력

구조적 요구사항:

- 새로운 결제 수단이 추가되더라도 Booking API의 핵심 비즈니스 로직 수정이 최소화되어야 한다.
- 결제 수단별 검증과 실행 로직은 인터페이스, 전략 객체, 정책 객체 등으로 분리하는 것이 적절하다.

### 4.5 장애 대응 및 예외 처리

#### Redis 장애

Redis 장애 시 fallback 전략과 그 근거를 제시해야 한다.

고려해야 할 내용:

- Checkout 조회 경로의 cache fallback과 Booking 쓰기 경로의 admission fallback은 분리해서 다뤄야 한다.
- Redis 장애가 초과 판매로 이어지면 안 된다.
- Redis 장애 시 무제한 DB 직접 예약 경쟁으로 전환하면 공정성과 DB 안정성이 모두 훼손될 수 있다.
- fallback 경로는 DB 쓰기 부하를 제한하고, 정합성과 공정성 요구사항을 함께 만족해야 한다.

#### 결제 실패

한도 초과, 승인 실패 등 결제 실패 케이스에 대한 대응 로직을 설계해야 한다.

필수 동작:

- 결제 실패 요청은 최종 확정 예약을 만들면 안 된다.
- 실패한 결제가 재고를 영구히 점유하면 안 된다.
- 선점된 재고는 선택한 admission 정책에 따라 해제되거나 다음 요청자가 사용할 수 있어야 한다.
- 결제 실패 응답도 멱등성 정책 안에서 재응답 가능해야 한다.

#### 애플리케이션 서버 장애

애플리케이션 서버는 2대 이상으로 구성된 분산 환경을 가정한다.

기대 동작:

- 한 인스턴스가 장애가 나면 트래픽은 살아 있는 인스턴스로 전달되어야 한다.
- 진행 중이던 예약/결제 요청은 멱등성 저장소, 예약 TTL, 영속 상태를 통해 복구 가능해야 한다.
- 비즈니스 정합성은 특정 JVM의 로컬 상태에 의존하면 안 된다.

#### DB 장애

DB는 예약, 결제, 재고의 최종 source of truth다.

기대 동작:

- DB 쓰기가 불가능한 상황에서는 최종 예약 성공 응답을 반환하면 안 된다.
- 서비스 불가 또는 재시도 가능한 실패 응답처럼 예측 가능한 방식으로 실패해야 한다.
- Redis 임시 선점, 결제 불확실 상태 등은 복구 또는 보상 가능해야 한다.

## 5. 구현 및 문서화 기대사항

프로젝트는 코드 수정 없이 실행 가능해야 한다. 추가 인프라 구성이 필요하다면 실행 방법과 필요한 설정을 명확히 문서화해야 한다.

문서에 포함하면 좋은 내용:

- 시스템 아키텍처
- 실행 방법
- API 목록과 예시
- 시퀀스 다이어그램 또는 플로우차트
- 주문/결제 도메인 중심의 ERD 또는 DDL
- 주요 기술적 판단과 선택 근거
- 라이브러리 도입 사유
- 장애 및 예외 상황에 대한 문제 해결 전략
